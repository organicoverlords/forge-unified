Do the following exactly and complete all steps before reporting.

OpenCode workflow requirements for this benchmark:

* Start by using the todo_write tool to create a visible checklist for all six phases.
* Update todos as work progresses. Mark a todo completed immediately after its evidence exists.
* Use the task tool at least once as an OpenCode-style subagent for broad repo exploration before doing direct search/grep.
* Use batch_parallel at least once for independent repo-inspection work, such as repo identity, file listing, config reads, and search.

# Phase 1 — Environment + repo inspection

1. Identify:

* current working directory
* git repo root
* current branch
* HEAD commit
* remote(s)
* dirty state
* build system
* package manager
* test commands
* repo size (rough estimate)
* top 15 largest files/directories

2. Create a concise repo map:

* major folders
* probable entrypoints
* config files
* tests
* scripts
* docs
* possible dead systems or duplicates

3. Find:

* TODO/FIXME/HACK comments
* disabled code
* stale branches of implementation
* duplicate functionality
* giant files (>1000 LOC)
* suspicious config drift

Do not stop after shallow inspection.

# Phase 2 — Long tool loop test

Choose ONE suspicious issue in the repo and investigate it deeply.

Required investigation loop:

1. inspect code
2. inspect related config
3. inspect logs/errors/build output
4. search for references/usages
5. form hypothesis
6. test hypothesis
7. revise if wrong
8. produce evidence

You must do at least **8 meaningful tool calls** during this phase.

Do not pretend certainty.

Label:

* VERIFIED
* LIKELY
* UNKNOWN

# Phase 4 — Judgment test

Choose ONE high-value but low-risk improvement.

Before editing anything:

* explain why this change matters
* estimate blast radius
* estimate implementation difficulty
* estimate rollback difficulty

Then implement the change.

Rules:

* no giant rewrites
* no unrelated cleanup
* preserve behavior
* must be provably useful

After implementation:

* validate with tests/build/lint if possible
* inspect diff
* summarize changed files

Before finishing:

Verify:

* no accidental secrets exposed
* no unrelated file changes
* no giant logs created
* temporary folder only contains expected files
* repo state is understood

Report:

* files created
* files deleted
* files modified
* tests run
* unresolved risks
