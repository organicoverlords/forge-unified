Use the available app tools for filesystem, shell, git, web/search, and read/write/remove operations inside the repo workspace.

Do the following exactly and complete all steps before reporting. Each phase must be backed by tool results.

Mandatory benchmark checkpoint:

- Do only minimal repo inspection before the file-operation phase.
- Before extended searching, write, read back, and remove the required `.agent_test` files using the app file tools.
- After that, perform the temporary `PROJECT_STATE.md` repo-file edit with a file tool, inspect `git diff` or `git status`, and decide keep or revert.
- Only then write the final Founder report and Technical report.

# Phase 1 — Environment + repo inspection

1. Identify current working directory, git repo root, current branch, HEAD commit, remotes, dirty state, build system, package manager, test commands, approximate repo size, and top 15 largest files/directories.

2. Create a concise repo map: major folders, probable entrypoints, config files, tests, scripts, docs, possible dead systems or duplicates.

3. Find TODO/FIXME/HACK comments, disabled code, stale branches of implementation, duplicate functionality, giant files over 1000 LOC, and suspicious config drift. Do not stop after shallow inspection.

# Phase 2 — Long tool loop test

Choose ONE suspicious issue in the repo and investigate it deeply.

You must perform at least 8 meaningful tool calls in this phase.

During the investigation, inspect relevant code, config, tests/scripts, usages/call sites, form a hypothesis, test it with at least one command or file read, revise if evidence contradicts it, and label important claims as VERIFIED, LIKELY, or UNKNOWN.

---

# Phase 3 — File operation stress test

Use app file tools, not shell-only shortcuts, for this phase.

Create `.agent_test/` by writing these exact files with the file write tool:

* `.agent_test/repo_summary.md`
* `.agent_test/investigation.md`
* `.agent_test/action_plan.json`

Then use the file read tool to read back all three files. Verify their contents in prose from the tool results.

Then remove only `.agent_test/investigation.md` with the app file removal tool.

Then verify `.agent_test/repo_summary.md` and `.agent_test/action_plan.json` still exist, and verify `.agent_test/investigation.md` is gone. You may use file_list or a bounded shell_command for this verification, but the write/read/remove operations themselves must be file tools.

# Phase 4 — Judgment test

Before editing real repo files, explain blast radius, difficulty, rollback plan, and why the change is low-risk.

Then implement ONE low-risk improvement based on the investigation. For this benchmark proof, make the improvement as a temporary documentation edit to `PROJECT_STATE.md` using a file edit or file write tool, then inspect `git diff`. After validation, decide whether to keep or revert. If you revert, use a file edit or file write tool so the repo ends clean.

After the edit, run the most relevant validation command available, inspect the diff, and decide whether to keep or revert the edit.

---

# Phase 5 — Final reports

Write two reports: a founder report of 180 words or less, and a technical report with exact steps taken, files touched, commands run, evidence, and unknowns.

---

# Phase 6 — Cleanup and safety

Before final answer, verify no private credentials were printed, no unrelated files changed, and no giant logs/artifacts were left in the repo. Report files created/removed/modified, validation commands run, remaining risks, and confidence score 0-100.
